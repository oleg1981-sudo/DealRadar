# Red-team register: URL/slug spec v1 → v2

Adversarial review of [v1-superseded.md](v1-superseded.md), run 2026-07-08.
Method: 6 hostile specialist agents (Postgres, Next.js, Netlify, integration, SEO, chaos) → adjudicator re-verified every 'confirmed' claim against files/docs. 49 raw findings → 25 accepted (1 blocker, 16 major, 8 minor), 1 rejected. Every accepted fix is incorporated in [spec.md](spec.md).

=== 1. [BLOCKER] Phase C→D window breaks every deal URL: Step B self-308 loops for backfilled rows, hard 404s for rows minted after the B backfill
CLAIM: The resolver ladder deploys in Phase C while all stored slugs are still old-format (rewrite is Phase D) and legacy_slug = slug (plan.md:29). Old-format segments miss Step A (regex-verified), hit Step B, and 308 'to current canonical' = the same URL → infinite redirect loop site-wide for the whole C window. Rows inserted during C (old deals_set_slug trigger still active; mint triggers arrive only in D1) have NULL public_id and NULL legacy_slug → their live homepage-card URLs fall through all three steps → hard 404. The unconditional freeze (NEW.public_id := OLD.public_id) then pins NULL forever, and D1's slug rewrite composes slug_base||'-d'||NULL → NOT NULL violation aborts the flip (or strands the cohort). The Phase C checkpoint ('legacy URL → 308 → 200 one hop') is mathematically unsatisfiable pre-D, falsifying plan.md:8's independent-deployability claim.
FIX: Spec §4 Step B (lines 85-88): rewrite to 'Step B — exact .eq(slug, raw) on the live slug column → 200 self-canonical (covers the pre-flip window); then exact .eq(legacy_slug, raw), then lower() variant → if row.slug === raw render 200, else 308 to current canonical.' Plan Phase B: move the mint triggers to B — BEFORE INSERT sets public_id := coalesce(NEW.public_id, substr(md5(NEW.product_id),1,12)) AND legacy_slug := coalesce(NEW.legacy_slug, NEW.slug); BEFORE UPDATE freezes NULL-safely (NEW.public_id := COALESCE(OLD.public_id, substr(md5(NEW.product_id),1,12))). Plan D1: add a pre-rewrite delta backfill (UPDATE deals SET public_id=…, legacy_slug=coalesce(legacy_slug,slug) WHERE public_id IS NULL OR legacy_slug IS NULL) plus the gate assertion count(*)=0 before the one-time slug UPDATE. Plan C checkpoint (line 42): change to 'old-format URL sampled from prod DB → 200 with self-canonical (no redirect) pre-flip'. Todo C2 acceptance: add the equality guard + phase-aware smoke assertions. Plan: add 'rollback floor' note — after D1, full URL compatibility requires the C build or newer (pre-C builds 404 previously-published old-format URLs, though on-site nav still works); hold C in prod ≥1 week before D.

=== 2. [MAJOR] Expiry re-activation trigger has no discriminator — both naive implementations of §5 are broken, and the unguarded historical-lows RPC compounds it
CLAIM: Spec §5 says the cron sets status='expired' and 'any later upsert re-activates (trigger clears status/expired_at)', but at trigger level the cron's UPDATE, the update_historical_lows_batch RPC's UPDATE (runs after every refresh, no value-change guard), and a writer's ON CONFLICT DO UPDATE are indistinguishable. If the trigger clears status on any UPDATE, the cron's own write is reverted (nothing ever expires); if it waits for writers to send status, it never fires (neither toRow() nor the ingest payload contains a status key, and PostgREST merge-duplicates touches only payload columns). Downgraded from blocker: D3's staging verification ('age a row, cron marks it, upsert re-activates') would catch either failure mode before prod, and the fix is a one-predicate spec addition.
FIX: Spec §5 line 99: replace 'Any later upsert re-activates (trigger clears status/expired_at)' with the exact predicate: in the BEFORE UPDATE trigger, IF NEW.last_updated IS DISTINCT FROM OLD.last_updated THEN NEW.status := 'active'; NEW.expired_at := NULL; END IF — writers always send a fresh last_updated; the expiry cron and update_historical_lows_batch never touch it. Add: 'writers MUST NOT send status/expired_at.' Todo D3 verify: add 'cron-expire a row → run updateHistoricalLows → assert still expired; re-upsert → assert active.'

=== 7. [MAJOR] Sitemap shard query: PostgREST max-rows (default 1000) silently truncates every shard, the new partial index doesn't serve the query, and mutable NULL-heavy ordering makes shard membership unstable
CLAIM: Three compounding defects. (1) Spec line 193's 'sharding fixes it regardless' is false: max-rows caps every response, so a 10k .range() window returns 1000 rows and rows 1000-9999 of each window appear in NO shard — silent ~90% loss, worse than today's visible 5000 cap. (2) The only new index is (last_updated DESC) WHERE status='active' (line 98) but the query orders by content_changed_at (line 106) — full scan+sort per shard render. (3) Phase B's backfill omits content_changed_at entirely, so the whole pre-migration catalog is NULL (which sorts NULLS FIRST under DESC), and a writer-bumped mutable sort key + OFFSET pagination reshuffles rows across shard boundaries between Google's separate fetches — duplicated and skipped URLs as steady state, breaking criterion 4.
FIX: Plan Phase B: add `content_changed_at := last_updated` to the backfill list. Spec §6: shard on an immutable key — keyset-paginate ORDER BY public_id with fixed ranges (keep lastmod = content_changed_at as a field, not the sort; if freshness bias is wanted, apply it within shard 0 only); add partial index matching whatever the query actually orders by, WHERE status='active'; paginate each shard fetch internally in ≤1000-row .range() pages (or raise max-rows as a documented deploy prerequisite). Delete the 'sharding fixes it regardless' sentence (the A2 probe stays). Todo C8: add a count-parity assertion — |sitemap URLs| equals the DB active count within tolerance, and expired rows are absent.

=== 8. [MAJOR] Sitemap entry shape is wrong twice: Google's hreflang-sitemap method needs a <url> per locale variant, and 10k-entry × 14-alternate shards (~23 MB) exceed Netlify's function response limits
CLAIM: Spec line 108 claims '<loc> = default-locale variant … is the documented Google pattern' — it is the opposite: Google's localized-versions doc requires a separate <url> element for EVERY variant, each carrying the full alternate set. As specced, the 12 non-default self-canonical URLs per deal appear in no sitemap. Separately, the shard size is set against the protocol limit (50k/50MB) not the host: ~2.3 KB/entry with 14 xhtml:link alternates × 10k ≈ 23 MB per per-request-generated response, past Netlify's 6 MB synchronous cap (and ~20 MB streaming). Downgraded from the attacker's 'blocker': on-page hreflang (already emitted by the deal page) preserves variant discovery/canonicalization, so this degrades rather than breaks indexing.
FIX: Spec §6 line 108: delete 'this is the documented Google pattern'. Choose ONE method and state it: (a) emit 13 <url> entries per deal each with the identical alternates cluster — then entries = active_deals × 13 and shard size must be host-derived (~2-2.5k entries/shard ≈ 5-6 MB); or (b) drop sitemap-level alternates entirely, list all 13 plain locale URLs, and rely on the on-page hreflang the deal page already emits (Google: don't mix methods). Write the chosen byte math into the spec; update success criterion 4 and C5 acceptance.

=== 9. [MAJOR] content_changed_at 'bumped by writers' is impossible with blind full-row upserts, and the bump criterion references an 'availability' column that does not exist
CLAIM: Both writers are blind upserters with no read of the existing row (deals.repo.ts:60; ingest-awin.cjs streamed REST batches), so a writer cannot know whether price/name changed: it either always sends now() — inflating lastmod on every refresh, the exact failure the design exists to avoid — or SELECT-before-upsert, doubling round trips and racing the concurrency-6 refresh fan-out. Delete 'add content_changed_at bump logic to both writers' from plan D2/todo D2. Drop 'availability' from the criterion or define the column first.

=== 10. [MAJOR] 48h expiry is justified against a fictional 15-min cadence — the deployed scheduler is daily with a top-100-per-cell persistence cap, so live deals mass-expire, rank-100 rows flap, and two missed crons expire the catalog
CLAIM: Spec lines 99/194 justify 48h via '15-min refresh + nightly ingest', but the only deployed scheduler is netlify/functions/refresh-deals.mts at '0 4 * * *' (daily; the 15-min text is an aspirational comment), and each run persists at most 100 deals per country×category (route.ts:53, registry slice). Document the steady-state active-set cap and rank-flapping, or raise the per-cell refresh limit.

=== 12. [MAJOR] Collision odds understated ~178×, and the 16-hex runbook mints a URL the spec's own parser can never route; a collision also becomes a deterministic nightly ingest outage

Fix the §1 example to a 12-hex id. Add ingest-failure alerting to the deliverables (also required by the expiry finding).

=== 13. [MAJOR] Brand-prefix predicate treats brand as an ILIKE pattern and compares in raw text space — wildcard brands corrupt suppression, accented brands mint permanently duplicated slugs, and it cannot be mirrored in TS
CLAIM: product_name NOT ILIKE brand || '%' (spec lines 61-62) has two defects: unescaped % / _ in free-text feed brands act as wildcards ('100% Pure' wrongly matches '1000W Pure Sine Inverter' → brand suppressed), and the test runs pre-normalization while output is unaccent/lowered — brand "L'Oréal" vs feed name "L'Oreal Paris …" fails the prefix test (é≠e) → permanent slug 'l-oreal-l-oreal-paris-…'. Add '100% Pure' and L'Oréal-class cases to the parity fixtures (spec Testing strategy bullet 1).

=== 15. [MAJOR] Mutable slug_base + permanently-cached 308s turns feed-title oscillation into client-side redirect loops and lets one brand backfill 308-churn the whole corpus
CLAIM: slug_base recomputes on any name/brand change (spec line 69) and drift heals via 308, which browsers cache indefinitely (RFC 7538). A title flip A→B→A leaves a user's browser with cached S1→S2 while the server now serves S2→S1 → ERR_TOO_MANY_REDIRECTS client-side with no server-side fix until cache eviction. A brand backfill (NULL→value on thousands of rows) rewrites every affected slug_base in one trigger pass → corpus-wide recanonicalization. The spec records no rationale for recompute-on-change over freeze-after-mint even though Step A already makes stale bases cosmetically harmless — the spec explicitly accepts cosmetic staleness for truncation/case variants (line 82).
FIX: Spec §3 line 69: put the choice to the reviewer as a Phase A decision with a stated default of FREEZE — slug_base computed at first mint, recomputed only by explicit admin action (cost: cosmetic staleness the spec already accepts). If recompute is kept: serve drift-healing as 307 and reserve 308 for the legacy layer, add oscillation hysteresis, and a churn-budget alert (>X% slug rewrites/day).
